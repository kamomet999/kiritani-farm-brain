# 03 アーキテクチャ（技術選定）

## アーキを決める要件（[01](01-requirements.md) より）

1. データが数ヶ月〜通年で増え、**消えない保存**が要る（端末ローカル閉じ込め不可）。
2. **多デバイス**（農場長のスマホ/PC、現場閲覧）で同じデータを見る。
3. **写真**の保存・表示、**AI**（テキスト生成＋vision）を扱う。
4. **月額をできるだけ¥0**に。現場スマホで使える。
5. 全体図ダッシュボードは**イラスト化・対話型**（クリックで動作）。

## 技術案の比較

| 案 | 内容 | 多デバイス/保存 | 写真・AI | 月額 | 判定 |
|---|---|---|---|---|---|
| A. 単一HTML＋localStorage | v0でやった形 | ✗ 端末内のみ | ✗ 実質不可 | ¥0 | 使い捨て検証のみ。本システムには不適 |
| B. PWA＋ローカルDB | オフライン強い | △ 同期に別基盤要 | △ | ¥0 | 単体では要件不足 |
| C. **Next.js＋Supabase＋Claude API** | Web/PWAアプリ＋クラウドDB＋ストレージ＋AI | ◎ | ◎（Storage＋vision） | ほぼ¥0（無料枠） | **推奨** |
| D. APEに相乗り | 既存の農業アプリ基盤を流用 | ◎ | ◎ | ほぼ¥0 | 7bayは独立方針。**パターン/スキーマは参考にする**が相乗りはしない |

## 推奨：C（Next.js 15 ＋ Supabase ＋ Claude API）

理由：
- **Supabase**＝Postgres（[02](02-data-model.md)のモデルをそのまま）＋認証＋**Storage（写真）**＋リアルタイム。無料枠が実用的。
- **多デバイス**：URLで農場長スマホ/PC・現場が同じデータ。PWA化で現場入力も快適。
- **AI**：サーバー側APIルートから **Claude API**（作業指示のテキスト生成）＋**vision**（生育写真解析）。マルチモーダルLLMは実在確認済。
- **全体図**：**SVGベースの対話型スキーマ図**（各ハウス=クリック可能な図形、状態でSVGの色/アイコンを差し替え）。イラスト化・圧縮配置に向く。
- 既存 **APE が同スタック**（Next.js15+Supabase）なので知見流用が効く（相乗りはせず、スキーマ/パターンを参考）。

## 各連携の方針

- **写真**：Supabase Storage にアップ → URLをPhotoに保存 → 必要時にvisionへ渡す。
- **AI作業指示**：朝の観察＋最新測定＋写真を1プロンプトに束ね、Claudeで生成 → AIInstructionに保存、農場長が採用/修正。
- **測定データ**：v1は**手入力**。機器連携（CSV取込 or API）は機種確定後（[06](06-decisions-open.md)）。実装は [orchestrator-mode] に従い委譲。
- **全体図ダッシュボード**：座標(map_x,map_y)を持つSVG。**航空写真を参照に一度だけ配置を作り**、以後はデータ駆動で状態色/アイコンを更新。

## この環境での注意（開発時）

- 本PCは **TLS中継**で npm/bun install が失敗しやすい（`NODE_TLS_REJECT_UNAUTHORIZED=0` 等で回避）。
- 本PCは **Docker/WSL不可**（BIOS仮想化無効）。Supabaseはクラウド（ローカルsupabase不要）、デプロイはVercelで回避。
- ローカル起動が難しければ、Vercelプレビューで検証する運用も可。
